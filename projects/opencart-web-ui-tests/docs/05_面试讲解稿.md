# OpenCart Web 自动化项目面试讲解稿

## 1. 这个项目你是怎么做的？

我基于 OpenCart 电商系统做了一个 Web UI 自动化回归测试项目。项目中选择了电商系统比较核心的用户路径，包括首页访问、登录页校验、错误登录、商品搜索、商品详情页校验、加入购物车和购物车页面校验。

技术上使用 Python + Selenium + pytest，采用 Page Object 模式封装页面元素和页面操作，用 pytest 组织测试用例，用 Allure 生成测试报告。测试失败时会自动截图，方便定位问题。

## 2. 为什么使用 Page Object？

UI 自动化测试中，页面元素定位比较容易变化。如果所有定位器都直接写在测试用例里，一旦页面变化，维护成本会很高。

Page Object 的作用是把页面元素和页面操作封装到页面类中。例如 HomePage 负责首页搜索，SearchPage 负责搜索结果页，ProductPage 负责商品详情页。这样测试用例只关心业务流程，页面变化时主要修改 Page 类即可。

## 3. 你怎么设计测试用例？

我先按电商核心流程拆分功能模块：

1. 首页是否能打开；
2. 登录页表单是否正常；
3. 错误账号是否无法登录；
4. 搜索有效关键词是否有结果；
5. 搜索不存在关键词是否无结果；
6. 从搜索结果进入商品详情页；
7. 商品详情页是否存在加入购物车按钮；
8. 加入购物车后购物车页面是否可访问。

其中前几个作为核心回归用例，购物车用例因为依赖公开 Demo 环境的 session 和页面状态，所以标记为 optional。

## 4. 你怎么处理 UI 自动化不稳定？

我主要从几个方面处理：

1. 使用显式等待，而不是大量 sleep；
2. 使用相对稳定的定位器，例如 name、CSS、XPath；
3. 使用 Page Object 集中维护定位器；
4. 对公开 Demo 环境不稳定的购物车流程标记 optional；
5. 测试失败时自动截图并附加到 Allure 报告；
6. 在文档中记录环境风险和处理决策。

## 5. 你为什么没有继续使用本地 OpenCart？

我最开始尝试过 Windows WSL2 + Docker Desktop 本地部署 OpenCart。过程中先遇到端口映射问题，后续通过修改宿主机暴露端口和减少不必要端口暴露解决。之后又遇到数据库表未初始化问题，我通过查看日志、检查数据库表、手动执行 CLI 安装脚本完成了数据库初始化。

但数据库初始化成功后，页面出现 Twig 模板加载异常。我判断 Docker 网络、端口、数据库连接、数据库初始化都已经排除，问题边界转移到了 OpenCart 当前分支代码或模板加载逻辑。

由于本项目目标是完成 Web 自动化测试实践，而不是修复 OpenCart 源码，所以我记录了环境问题，并切换到 OpenCart 官方 Demo 环境继续完成自动化测试。在 WSL2 中使用 Selenium + Chrome Headless 访问 `https://demo.opencart.com` 时，浏览器可以正常启动，但页面标题返回：Just a moment... 这说明 Selenium 和 Chrome 环境本身正常，问题发生在被测站点侧：OpenCart 官方 Demo 对自动化浏览器返回了 Cloudflare / 防机器人验证页面。
处理决策：
1. 不尝试绕过 Cloudflare 或验证码；
2. 记录该公开测试环境风险；
3. 将被测地址切换为 `https://tutorialsninja.com/demo/`；
4. 继续完成 OpenCart 电商系统 Web UI 自动化测试项目。
该处理方式体现了测试过程中对外部环境风险的识别、记录和规避。

## 6. 你这个项目体现了哪些测试能力？

这个项目主要体现：

1. 能根据业务流程设计测试用例；
2. 能使用 Selenium 编写 Web UI 自动化脚本；
3. 能使用 pytest 管理测试用例和 fixture；
4. 能使用 Page Object 提高脚本可维护性；
5. 能使用 Allure 生成测试报告；
6. 能在失败时保留截图和日志；
7. 能记录环境问题并进行风险取舍；
8. 能用 Git 管理项目并通过 SSH 上传 GitHub。
